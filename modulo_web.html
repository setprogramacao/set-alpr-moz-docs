Módulo Desktop — Arquitectura, Funcionamento e Integração

1. Introdução

O módulo web do sistema ALPR UNIPIAGET constitui o núcleo de gestão centralizada da plataforma. Desenvolvido sobre a biblioteca FastAPI (Ramírez, 2018), este módulo assume três responsabilidades principais: (i) expor uma interface de programação de aplicações (API RESTful) que recebe as deteções enviadas pelo módulo desktop; (ii) processar e persistir os registos de acesso numa base de dados relacional; e (iii) disponibilizar ao utilizador um painel de administração acessível por navegador web, que apresenta estatísticas, histórico de movimentos, relatórios e ferramentas de gestão.

A separação em módulos independentes — desktop e web — segue o princípio arquitectónico de separação de preocupações (separation of concerns), pelo qual cada componente do sistema é responsável por um conjunto coeso e delimitado de tarefas (Martin, 2017). O módulo web não realiza captura de imagem nem processamento de visão computacional; limita-se a receber dados estruturados do módulo desktop e a expô-los de forma organizada para os utilizadores administrativos.

A entidade RegistroAcesso aceita veiculo_id nulo de forma intencional: quando o sistema reconhece uma placa que não se encontra cadastrada na base de dados, o registo é criado igualmente, vinculando apenas a matrícula lida e o momento da passagem. 4.1 Lógica de Toggle Entrada/Saída

Uma das funcionalidades mais relevantes do serviço de deteção é a determinação automática do tipo de movimento (entrada ou saída). Em vez de depender do valor enviado pelo módulo desktop — que transmite sempre "entrada" como marcador de posição — o servidor executa a seguinte lógica:

  1. Consultar o último registo existente para a placa recebida.
  2. Se não houver registo anterior, o tipo é definido como entrada.
  3. Se o último registo for entrada, o próximo será saída, e vice-versa.

Este mecanismo garante que o sistema mantém coerência no historial de movimentos, independentemente do estado interno do módulo desktop. A responsabilidade sobre o tipo de movimento reside exclusivamente no servidor, garantindo fonte única de verdade (single source of truth).

4.2 Deteção de Duplicatas

Para evitar registos redundantes provocados por deteções repetidas da mesma placa num curto intervalo de tempo (por exemplo, o veículo demora a atravessar o portão), o serviço verifica se já existe um registo para a mesma matrícula nos últimos N minutos (por omissão, 5 minutos, valor configurável na tabela Configuracao). Caso seja identificada uma duplicata, a resposta inclui o campo duplicata: true e o tempo decorrido desde o último registo, mas não cria um novo registo na base de dados.

5. API RESTful — Endpoints Disponíveis

A interface de programação de aplicações do módulo web segue os princípios REST definidos por Fielding (2000), onde os recursos são identificados por URI e manipulados por verbos HTTP normalizados (GET, POST, PUT, DELETE). A documentação interactiva é gerada automaticamente pelo FastAPI em /docs (Swagger UI) e /redoc (ReDoc).

6. Autenticação e Autorização

O acesso ao painel web é protegido por um mecanismo de autenticação baseado em JSON Web Tokens (JWT), conforme especificado por Jones et al. (2015). A palavra-passe é armazenada em formato irreversível através do algoritmo bcrypt, que incorpora um valor aleatório de sal (salt) em cada hash, tornando ataques por dicionário ineficazes mesmo em caso de comprometimento da base de dados (Provos & Mazières, 1999).

Figura 5 — Fluxo de autenticação JWT e hierarquia de níveis de acesso.

O token JWT é armazenado no localStorage do navegador e enviado em cada requisição subsequente no cabeçalho HTTP Authorization: Bearer <token>. O servidor valida a assinatura criptográfica do token a cada pedido, sem necessidade de consultar a base de dados, o que permite escalabilidade horizontal (Fielding, 2000). O token expira ao fim de 24 horas, após o qual o utilizador deve autenticar-se novamente.

7.1 Visualizações do Dashboard

O dashboard (página inicial após autenticação) é a principal ferramenta operacional do sistema. Apresenta os seguintes componentes visuais, todos actualizados por chamadas assíncronas à API REST:

  • Cartões de KPI — total de veículos cadastrados, total de proprietários, ocupação actual do campus, e número de entradas e saídas do dia.
  • Gráfico de barras — actividade horária — entradas e saídas agrupadas por hora nas últimas 24 horas, renderizado com Chart.js (Chart.js, 2023).
  • Gráfico de linha — tendência semanal — total de movimentos por dia nos últimos 7 dias.
  • Gráfico circular — distribuição por categoria — proporção de docentes, alunos, técnicos e visitantes nos registos do período.
  • Lista de veículos no campus — matrículas com entrada registada e sem saída correspondente, com o tempo de permanência calculado.
  • Alertas — docentes ausentes por mais de 3 dias consecutivos e permanências anormalmente curtas (menos de 1 hora).

7.2 Exportação de Dados

O módulo web oferece exportação de registos de acesso e relatórios de proprietários em formato CSV (Comma-Separated Values), compatível com Microsoft Excel e LibreOffice Calc. A exportação é gerada dinamicamente pelo servidor e transferida directamente para o dispositivo do utilizador através de um cabeçalho HTTP Content-Disposition: attachment.

8. Comunicação com o Módulo Desktop

A comunicação entre o módulo desktop e o módulo web ocorre de forma unidirecional: o desktop envia dados, o web responde. Não existe comunicação no sentido inverso (o servidor não aciona comandos no módulo desktop). 8.1 Exemplo de Payload

O exemplo seguinte ilustra uma transação típica entre o módulo desktop e o módulo web, após o sistema OCR identificar a matrícula MP123AB com 95% de confiança:

POST /api/v1/registros/deteccao HTTP/1.1 Host: 192.168.1.100:8000 Content-Type: application/json { "placa_detectada": "MP123AB", "tipo_movimento": "entrada", "confianca_ocr": 0.95, "metodo_ocr": "hibrido", "imagem_base64": "iVBORw0KGgoAAAANSUhEUgAA...", "timestamp": "2025-03-11T14:30:00" } HTTP/1.1 201 Created Content-Type: application/json { "sucesso": true, "mensagem": "Detecção registada [ENTRADA] — Dr. João Silva (docente)", "registro_id": 145, "veiculo_cadastrado": true, "veiculo": { "placa": "MP123AB", "modelo": "Toyota Corolla", "proprietario": { "nome": "Dr. João Silva", "categoria": "docente", "departamento": "Engenharia Informática" } }, "duplicata": false, "tempo_desde_ultimo_registro": null }
Nota: O campo tipo_movimento no payload de requisição é ignorado pelo servidor. O módulo web determina autonomamente se o movimento é entrada ou saída, com base no historial de registos daquela matrícula, garantindo consistência independente do estado do módulo desktop.

11. Visão Global — Integração com os Demais Módulos

O módulo web posiciona-se como a camada de gestão e persistência da arquitectura global do sistema. A Figura 7 apresenta a relação entre os três módulos principais do projecto ALPR UNIPIAGET, com os protocolos de comunicação utilizados em cada interface.

Figura 7 — Diagrama de integração entre os três módulos do sistema ALPR UNIPIAGET.

O módulo web é o único com acesso directo à base de dados. Os módulos desktop e shared não escrevem na base de dados diretamente — toda a persistência ocorre exclusivamente através das chamadas à API REST do módulo web. Esta arquitectura centraliza a integridade dos dados e facilita a escalabilidade futura do sistema, nomeadamente a adição de novos clientes (aplicações móveis, outros postos de controlo) sem necessidade de alterar a lógica de negócio.

12. Considerações de Segurança

O desenvolvimento do módulo web incorporou as seguintes práticas de segurança recomendadas para sistemas de informação em ambiente universitário:

  • Palavras-passe com bcrypt — O algoritmo bcrypt é resistente a ataques de força bruta por ser computacionalmente custoso por design; o factor de custo pode ser aumentado à medida que o hardware evolui (Provos & Mazières, 1999).
  • CORS configurável — Apenas origens explicitamente autorizadas podem interagir com a API a partir de um navegador, prevenindo ataques de falsificação de pedidos entre sítios (cross-site request forgery).
  • Validação de dados com Pydantic — Todos os dados recebidos pela API são validados contra schemas rígidos antes de chegarem à camada de serviços, prevenindo injeção de dados malformados.
  • ORM com queries parametrizadas — O SQLAlchemy utiliza consultas parametrizadas internamente, eliminando o risco de injeção SQL.
  • Rastreabilidade via LogAuditoria — Todas as acções relevantes dos utilizadores são registadas com data e hora, permitindo auditoria posterior em caso de incidente.
  • Endpoint de deteção sem autenticação — O endpoint POST /registros/deteccao não requer autenticação por operar exclusivamente em rede local (LAN do campus). Em implantações expostas à Internet, recomenda-se a adição de uma chave de API estática.
